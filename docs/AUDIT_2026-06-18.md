## 3. Remaining work — TypeScript cleanup

**Scope:** `packages/server` only. `packages/web` is clean (0 errors). `packages/engine` is clean.

**Why it matters:** tsx bypasses tsc so the server runs fine today. But 76 silent errors mask real future bugs — the type system provides no protection when this many are suppressed.

### Root cause A — @types packages not hoisted by npm workspaces

`@types/pg`, `@types/bcryptjs`, `@types/pdfkit` are listed in `packages/server/package.json` devDependencies but npm workspaces did **not** hoist them to the root `node_modules/@types/`. They are absent from the filesystem, causing ~3 "implicit any" errors on third-party imports.

**Fix — run from repo root:**
```bash
npm install --save-dev @types/pg@^8.11.0 @types/bcryptjs@^2.4.6 @types/pdfkit@^0.13.4
```
(Root-level install ensures hoisting works correctly across all workspace packages.)

### Root cause B — `puppeteer` and `svg-to-pdfkit` not installed

`puppeteer` is imported in `src/routes/estimates.ts` and `src/scripts/generate_proposal_local.ts` but is not in any `package.json`. `svg-to-pdfkit` is imported in `src/utils/pdf-proposal-kit.ts` but also absent.

**Fix:**
```bash
# puppeteer bundles its own types — no @types needed
npm install --save-optional puppeteer -w packages/server

# svg-to-pdfkit (no bundled types, check npm for @types or declare module)
npm install svg-to-pdfkit -w packages/server
# If no @types/svg-to-pdfkit exists, add to packages/server/src/types/svg-to-pdfkit.d.ts:
# declare module 'svg-to-pdfkit';
```

### Root cause C — `EngineEstimate` type not imported in `estimates.ts`

`estimates.ts` line 304 uses `EngineEstimate` as a type annotation but the import at line 6 only brings in `calculateEstimate` and `VisibilityProfile`:

```ts
// current (line 6):
import { calculateEstimate, type VisibilityProfile } from '@es/engine';

// fix — add the type alias:
import { calculateEstimate, type VisibilityProfile, type Estimate as EngineEstimate } from '@es/engine';
```

This also fixes the cascading `Map<unknown, unknown>` errors at lines 343 and `services/estimate-calculation.ts:106` because the materialMap callback `m` will be inferred correctly once the `EngineEstimate` type resolves.

Also in `utils/visibility.ts` line 1 — `EngineEstimate` is imported but never used. Remove it:
```ts
// remove EngineEstimate from that import
```

### Root cause F — Implicit `any` on callback parameters

The drizzle query results are typed `any[]` in several places, causing implicit any when iterating. Fix by typing the callback parameter explicitly.

**`routes/dashboard.ts`** — ~8 errors:
```ts
// pattern:
allEstimates.filter(e => ...) // e is any

// fix — import the Drizzle inferred type or cast inline:
const allEstimates = await db.select().from(schema.estimates)...;
type EstimateRow = typeof allEstimates[number];
allEstimates.filter((e: EstimateRow) => ...)

// or simpler — cast in the sort:
.sort((a: typeof allEstimates[number], b: typeof allEstimates[number]) => ...)
```

Affected lines: 60, 80, 83, 84, 85, 103, 112, 124 (a, b in sort).

**`routes/estimates.ts`** — ~10 errors:  
Lines 91 (c), 93 (est), 136 (m), 282 (m), 310 (l), 324 (p), 333 (s), 356 (s, l), 521 (s), 607 (m), 608 (l), 914 (m), 916 (layer).

**`routes/materials.ts`** — line 40 (mat).

**`routes/users.ts`** — line 46 (user).

**`services/estimate-calculation.ts`** — lines 34 (m), 66 (l), 80 (p), 89 (s), 128 (s).

Pattern for all of them — use `typeof queryResult[number]` inline or extract a type alias:
```ts
const materials = await db.select().from(schema.materials).where(...);
type MatRow = typeof materials[number];
const materialMap = new Map(materials.map((m: MatRow) => [m.id, { ... }]));
```

### Root cause G — `costPerM2` property access on untyped layer

`routes/estimates.ts:919` in `requoteEstimateRoute`:
```ts
const mat = materialMap.get(layer.materialId); // mat is {} | undefined
const newCostPerSqM = mat?.costPerM2; // TS2339: Property 'costPerM2' does not exist
```

`costPerM2` is a calculated value from the engine, not a DB column. The fix depends on intent:
```ts
// if it's meant to be computed on the fly:
const newCostPerSqM = mat
  ? (parseFloat(mat.costPerKgUsd) * parseFloat(layer.micron) * (mat.density ?? 1)) / 1000
  : 0;

// or if sourced from prior calculation, use the layer's stored value
```

### Root cause H — `templates.ts` route registration type mismatch (lines 237–238)

The `registerTemplateRoutes` function passes a generic `FastifyRequest` to handlers that expect typed requests with `Params` and `Body`. This is a structural issue with how Fastify route handlers are wired:

```ts
// current — causes TS2345:
fastify.post('/api/v1/templates/:id/instantiate', async (request, reply) =>
  instantiateTemplateRoute(fastify, request, reply));

// fix — add explicit generic to the route:
fastify.post<{ Params: { id: string }; Body: { customerId?: string; jobName?: string } }>(
  '/api/v1/templates/:id/instantiate',
  async (request, reply) => instantiateTemplateRoute(fastify, request, reply)
);
```

Apply same pattern to the `getTemplateByIdRoute` registration.

### Root cause I — Self-referential `estimates` table (structural, low priority)

`schema.ts:96` — `estimates.sourceEstimationId` FK references `estimates.id` in its own initialiser. TypeScript resolves the type as `any` (TS7022). This is a known Drizzle ORM limitation with circular FKs.

Fix: add an explicit type annotation to the table definition or suppress with `// @ts-expect-error -- circular FK`:
```ts
// @ts-expect-error -- self-referential FK, Drizzle limitation
export const estimates = pgTable('estimates', { ... });
```

Also `schema.ts:139` — a function with implicit return type caused by the circular reference chain. Add an explicit `: string | null` return type annotation to the function.
